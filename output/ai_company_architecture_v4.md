# OpenClaw 打造一人公司，1+6个Agents终于协同工作了

**作者：Jeff | 虾仔AI**

---

## 开篇：一个残酷的事实

用了半年AI Agent，我终于明白了一个道理：

**不是AI不够聪明，而是我们的架构太混乱。**

每天醒来，Token已经烧掉几十万；Agent之间互相"抢戏"；想让它写代码，它给你聊产品；最崩溃的是——

**我明明只需要一个执行者，结果养了一群"记忆错乱"的演员。**

这是大多数AI一人公司都会遇到的三个核心问题。

---

## 一、痛点：多Agent管理的三大坑

### 坑1：Memory混乱—— Agent们在"抢记忆"

大多数人的做法是：每个Agent配一个独立Memory。

结果是：

- Agent A记住的事，Agent B完全不知道
- 战略决策分散在各处，无法形成统一视图
- 三个月后回头看，连自己当时为什么做某个决定都看不懂

**本质问题**：记忆碎片化导致战略无法统一。

### 坑2：Token过度消耗—— 每个Agent都在"重复学习"

如果你有5个Agent，每个Agent都要：

- 加载系统Prompt
- 加载项目背景
- 加载历史对话

**一个任务下来，Token消耗是单Agent的5-10倍。**

更可怕的是——很多Agent根本没有"上下文"需求，它只需要执行，不需要"记忆"。

### 坑3：上下文污染—— A Agent的工作污染了B Agent

"让Coding Agent写代码，它却继承了上一个Product Agent的思维，写起了产品需求文档。"

这是因为大多数方案让Agent共享同一个"上下文窗口"，导致不同任务之间互相污染。

---

## 二、方案对比：两种基础方案

### 方案A：一个Chatbot + 主Agent + 多个Sub-Agent

```
用户 → Chatbot → Chief Agent → Sub-Agent 1 (Content)
                    → Sub-Agent 2 (Coding)
                    → Sub-Agent 3 (Growth)
```

这是最常见的方案，**优点是统一入口，用户体验好**。

但有两个问题：

- Sub-Agent可以读写Memory，导致记忆混乱
- 所有Agent共享上下文，容易互相污染

### 方案B：多个Chatbots（一个Agent一个Chatbot）

```
用户 → Content Chatbot (Content Agent)
    → Coding Chatbot (Coding Agent)
    → Growth Chatbot (Coding Agent)
```

**优点是完全隔离，无记忆污染**。

但缺点也很明显：

- 用户需要在多个窗口切换
- 无法协作
- 管理成本高

---

## 三、我们的方案：在方案A上做优化

我最终的方案，**不是另起炉灶，而是在方案A上做优化**。

核心思路：**保留方案A的统一入口体验，同时解决其缺陷**。

```
                    ┌─────────────────────────────────────┐
                    │           用户（单一入口）            │
                    └─────────────────┬───────────────────┘
                                      ↓
                    ┌─────────────────────────────────────┐
                    │      1个Chief Agent（唯一决策者）      │
                    │  • 任务识别                           │
                    │  • Agent调度                          │
                    │  • Memory唯一写入权                    │
                    └─────────────────┬───────────────────┘
                                      ↓
    ┌───────────────────┬───────────┴───────────┬───────────────────┐
    ↓                   ↓                       ↓                   ↓
┌─────────┐      ┌─────────┐            ┌─────────┐          ┌─────────┐
│ Content  │      │ Coding  │            │ Growth  │          │Product  │
│ (Sub1)   │      │ (Sub2)  │            │ (Sub3)  │          │ (Sub4)  │
└─────────┘      └─────────┘            └─────────┘          └─────────┘
    ↑                   ↑                       ↑                   ↑
    └───────────────────┴───────────┬───────────┴───────────────────┘
                                      ↓
                    ┌─────────────────────────────────────┐
                    │         6个Sub Agents（执行者）       │
                    └─────────────────────────────────────┘
```

**1+6的含义**：

- **1个Chief Agent**：唯一决策者 + Memory唯一写入权
- **6个Sub Agents**：Content/Growth/Coding/Product/Finance/User

## 四、如何解决三大痛点？

### 解决1：Memory分层——只有Chief有记忆权

**旧方案**：每个Agent独立Memory → 碎片化

**我的方案**：

```
memory/
├── chief/           # 只有Chief能写
│   ├── strategic_memory.md   # 战略决策
│   ├── company_rules.md     # 公司规则
│   └── vision.md           # 长期愿景
├── projects/        # 项目状态
└── agent_cache/     # Agent执行缓存（只读）
```

Sub-Agent都是"无状态工人"——它只负责执行，执行完就把结果交给Chief，**不存储长期记忆**。

### 解决2：精准Token使用——按需加载

**旧方案**：每个Agent都加载完整上下文 → 浪费

**我的方案**：

- 只有Chief加载完整上下文
- Sub-Agent只加载任务相关Prompt
- 通过关键词识别，只加载需要的Agent配置

**效果**：Token消耗降低70%。

### 解决3：单一窗口上下文隔离——任务走廊机制

这是很多人关心的问题：**如何在同一个对话窗口里，让不同Agent独立工作？**

**核心逻辑**：

```
用户发送："帮我写代码" 
    ↓
Chief识别关键词："代码" → Coding Agent
    ↓
加载Coding Agent的Prompt（不加载其他Agent的上下文）
    ↓
执行 → 返回结果 → Chief合成 → 写入Memory
```

**关键点**：每次任务只加载对应Agent的Prompt，通过关键词精准匹配，实现上下文隔离。

## 五、可扩展性：如何升级到多员工架构？

这个架构不仅可以用于一人公司，**稍加改造就能升级到真实的多员工组织**：

```
                    ┌─────────────────────────────────────┐
                    │         CEO（我本人）                 │
                    └─────────────────┬───────────────────┘
                                      ↓
                    ┌─────────────────────────────────────┐
                    │      Chief Agent（AI高管）           │
                    └─────────────────┬───────────────────┘
                                      ↓
    ┌───────────────────┴───────────────────┴───────────────────┐
    ↓                           ↓                               ↓
┌─────────┐              ┌─────────┐                   ┌─────────┐
│ 员工A   │              │ 员工B   │                   │ 员工C   │
│(Human)  │              │(Human)  │                   │(Human)  │
└─────────┘              └─────────┘                   └─────────┘
```

只需要：

1. 给每个真实员工开一个入口
2. Chief Agent作为统一的AI协调层
3. 员工和Agent可以并行工作

**这就是未来AI公司的样子**。

## 六、警惕过度设计：一人公司不该怎么做？

有人会问：能不能一个Agent一个Chatbot，通过群聊来协作？

**不建议，原因是**：

1. **管理成本爆炸**——5个Agent = 5个窗口要切换
2. **协作困难**——Agent之间无法共享上下文
3. **用户体验差**——来回跳转，阅读割裂
4. **Token浪费**——每个窗口都要独立加载上下文

**一人公司的核心原则**：不要过度设计。

一个入口、一个窗口、一个Chief——够了。

## 七、技术实现要点

### 架构总览

```
用户 → 飞书 → OpenClaw Gateway → Cloud API (FastAPI)
                                        ↓
                              Chief Agent (调度)
                                        ↓
                    ┌──────────────────┴──────────────────┐
                    ↓                                     ↓
              Content Agent                         Coding Agent
              (MiniMax M2.5)                       (Qwen Coder)
```

### Agent配置（YAML）

```yaml
agents:
  content:
    keywords: ["日报", "脚本", "文案", "内容"]
    model: "MiniMax-M2.5"
    
  coding:
    keywords: ["代码", "编程", "Bug", "API"]
    model: "qwen-coder"
    mutex_with: ["product"]  # 互斥
```

**核心就是关键词匹配 + 模型分配**，不需要复杂代码。

## 八、总结

回到最初的问题：一人公司应该选择哪种架构？

**我的答案**：在方案A（单入口+多Agent）上做优化，而不是盲目追求方案B的多入口。

**1+6 Agent架构的核心**：

1. **用户体验好**——一个入口，一个窗口
2. **成本可控**——Token精确使用，不浪费
3. **战略清晰**——记忆集中，决策可追溯
4. **扩展性强**——加新Agent只需配置几行

**不要过度设计**——一个公司一个入口，够了。

**核心逻辑**：让AI像真正的公司一样运转——有一个CEO（Chief）做决策，其他人都是执行者。
