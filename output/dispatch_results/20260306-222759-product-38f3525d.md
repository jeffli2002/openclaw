# Chief 自动派发机制 PRD 大纲

## 1. 产品概述

### 1.1 产品背景

Chief Agent 是整个 Agent 系统的决策中枢，负责接收用户任务并自动派发给对应的专业 Sub Agent 执行。当前已实现基础的分发能力，但需要将其产品化、规范化，形成完整的自动派发机制。

### 1.2 产品目标

| 目标类型 | 具体描述 | 优先级 |
|---------|---------|--------|
| 核心目标 | 实现用户消息到 Sub Agent 的自动路由与派发 | P0 |
| 效率目标 | 任务分发延迟 < 2秒，结果回传 < 60秒 | P1 |
| 稳定性目标 | 分发成功率 > 95%，失败可降级为 Chief 自执行 | P1 |
| 扩展目标 | 支持新增 Agent 类型和新任务类型热插拔 | P2 |

## 3. 核心功能

### 3.1 消息分类器

| 功能 | 描述 | 输入 | 输出 |
|-----|------|------|------|
| 关键词匹配 | 根据 `agent_keyword_router.yaml` 匹配关键词 | 用户消息文本 | 候选 Agent 列表 + 置信度 |
| 意图识别 | 识别消息是否为任务类（非闲聊/打招呼） | 用户消息文本 | is_task: true/false |
| 复杂度评估 | 评估任务复杂度，判断是否需要派发 | 用户消息文本 | complexity: simple/medium/complex |
| 长任务标识 | 识别耗时较长的任务，标记为异步处理 | 用户消息文本 | async: true/false |

### 3.2 任务派发器

| 功能 | 描述 | 输入 | 输出 |
|-----|------|------|------|
| 派发决策 | 根据分类结果决定派发策略 | 分类结果 + worker 配置 | delegate_spawn / fallback / chief_self |
| Worker Spawn | 启动一次性 Sub Agent worker | spawn_request | worker session_id |
| 派发日志 | 记录每次派发的详细信息 | 派发请求 + 结果 | 日志写入 memory/daily/ |
| 超时管理 | 监控任务执行超时 | 任务ID + 超时阈值 | 超时中断 + 降级处理 |

### 3.3 结果回传桥

| 功能 | 描述 | 输入 | 输出 |
|-----|------|------|------|
| 结果文件桥 | Worker 写入结果到指定文件 | dispatch_id | result_file 路径 |
| 结果等待 | Chief 等待 Worker 完成并读取结果 | result_file 路径 | 结果内容 |
| 结果整合 | 将 Worker 结果整理为统一格式 | 原始结果 | 格式化回复 |
| 失败处理 | 处理 Worker 执行失败的情况 | error 信息 | 降级执行或错误说明 |

### 3.4 配置管理

| 功能 | 描述 | 配置文件 |
|-----|------|---------|
| 关键词路由配置 | 定义各 Agent 的关键词映射 | `config/agent_keyword_router.yaml` |
| Worker 配置 | 定义各 Agent 的 worker 参数 | `config/chief_dispatch_workers.yaml` |
| Cron 任务配置 | 定义定时任务到 Agent 的映射 | `config/cron-agent-dispatch.yaml` |
| 降级规则配置 | 定义各场景的降级执行规则 | `config/chief_fallback_rules.yaml` |

### 3.5 监控与可观测性

| 功能 | 描述 |
|-----|------|
| 派发统计 | 记录每日派发次数、成功率、平均耗时 |
| 错误追踪 | 记录派发失败、Worker 错误、超时等异常 |
| 性能监控 | 监控派发延迟、Worker 启动时间、结果回传时间 |
| 告警通知 | 失败率超过阈值时通知 Chief 关注 |

## 4. 技术架构

### 4.1 核心组件

```
用户消息
    │
    ▼
┌─────────────────┐
│   消息分类器     │ ◄── config/agent_keyword_router.yaml
│  (agent_keyword │
│   _router.py)   │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│   任务派发器     │ ◄── config/chief_dispatch_workers.yaml
│  (chief_dispatch│
│      .py)       │
└────────┬────────┘
         │
    ┌────┴────┐
    │         │
    ▼         ▼
┌───────┐  ┌───────┐
│Spawn  │  │降级   │
│Worker │  │执行   │
└───┬───┘  └───┬───┘
    │          │
    ▼          ▼
┌─────────────────┐
│   结果回传桥     │
│(wait_dispatch   │
│ _result.py)     │
└────────┬────────┘
         │
         ▼
    用户收到回复
```

### 4.2 当前已实现模块

| 模块 | 状态 | 说明 |
|-----|------|------|
| 关键词路由 | ✅ 已完成 | `agent_keyword_router.py` + YAML 配置 |
| Worker 派发 | ✅ 已完成 | `chief_dispatch.py` 调用 `sessions_spawn` |
| 结果回传桥 | ✅ 已完成 | `wait_dispatch_result.py` + dispatch_results/ |
| 群聊绑定 | ✅ 已完成 | runtime `bindings` 配置 |
| Cron 调度 | ✅ 已完成 | `cron_dispatcher.py` + YAML 配置 |
| 降级执行 | ✅ 已完成 | Chief 按领域规则自执行 |

## 5. 验收标准

### 5.1 功能验收

| 场景 | 验收条件 |
|-----|----------|
| 私聊关键词派发 | 发送"写公众号"类消息，30秒内 Content Agent 开始执行 |
| 私聊打招呼不派发 | 发送"hi"或"在吗"，Chief 直接回复，不触发派发 |
| 群聊直接路由 | 在对应群聊发消息，Sub Agent 响应时间 < 5秒 |
| 定时任务派发 | Cron 触发任务，Sub Agent 正确接收并执行 |
| 结果回传 | Worker 执行完成后，用户收到整合后的结果 |
| 降级处理 | Worker 异常时，Chief 能降级执行或返回说明 |

### 5.2 性能验收

| 指标 | 目标值 |
|-----|--------|
| 派发延迟 | < 2秒 |
| Worker 启动时间 | < 10秒 |
| 结果回传时间 | < 60秒 |
| 派发成功率 | > 95% |

### 5.3 稳定性验收

| 场景 | 验收条件 |
|-----|----------|
| Worker 失败 | 记录错误日志，触发降级，消息告知用户 |
| 超时处理 | 30秒未完成，返回超时说明 |
| 配置错误 | YAML 格式错误时给出明确错误提示，不崩溃 |
