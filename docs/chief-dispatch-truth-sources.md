- `/root/.openclaw/workspace/config/agent_keyword_router.yaml`

- Validates and reads worker result bridges.
- Current default protocol: JSON.

## Current real execution flow

```text
Chief receives a private-chat task
-> simple question: Chief handles directly
-> domain task: chief_dispatch.py creates a plan
-> delegate_spawn: sessions_spawn(mode=run)
-> worker writes JSON result bridge
-> Chief reads result via wait_dispatch_result.py
-> if dispatch fails: Chief degraded-mode execution
```

## Operational hard gate (2026-03-10)

- For any private-chat **non-frontdesk** task, Chief must not skip the dispatch planner and start doing domain work directly.
- If planner result is `delegate_spawn`, Chief must perform a **real** `sessions_spawn` before presenting the work as handled by a domain worker.
- If dispatch is unavailable or fails, Chief may degrade to self-execution **only after** explicitly treating it as degraded mode.
- No fake delegation: if no real `sessions_spawn` / `sessions_send` happened, do not claim the task was already handed to Content / Growth / Coding / Product / Finance.

## Important clarification: why `agentId=main` is still valid

- In the current Feishu private-chat runtime, one-shot worker dispatch is implemented as **Chief-spawned subagent workers**.
- Therefore `config/chief_dispatch_workers.yaml` intentionally uses `runtime.agent_id=main`.
- The specialization comes from the worker prompt generated by `chief_dispatch.py` plus the worker config (`content` / `growth` / `coding` / `product` / `finance`), not from switching the spawned child session to a different runtime agent id.
- So seeing `agentId=main` in the spawn request is **not** evidence that dispatch is broken.
- The real failure pattern is: Chief recognized a domain task but still handled it directly without running planner/spawn.
