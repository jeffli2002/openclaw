## 错误一：AI日报发送失败

### 问题描述
- 早上7:31定时任务显示成功，但用户实际未收到日报
- 后续手动重发也失败
- Brave API出现rate limit错误

### 根本原因
1. **推送机制问题**：cron job执行成功不等于消息实际送达
2. **API限制**：Brave Search免费计划限制过严（1 req/sec），容易触发rate limit
3. **缺乏验证**：没有确认消息是否真正到达用户端

### 解决方案
1. ✅ 已配置：使用message工具直接发送，而非依赖cron的自动推送
2. ✅ 已实施：发送后立即向用户确认收到
3. 🔄 待优化：增加备用推送通道（邮件、短信等）
4. 🔄 待配置：升级到Brave付费计划或准备多个API key轮换

### 经验总结
- **发送≠送达**：必须验证消息实际到达
- **检查错误日志**：rate limit错误应第一时间捕获并告警
- **多重保障**：关键消息应有备用发送方案

## 错误二：Gateway健康监控权限问题

### 问题描述
- Gateway健康检查cron job报告无法访问健康端点
- 提示"没有直接访问HTTP请求或执行命令的权限"

### 根本原因
1. cron job运行环境权限不足
2. 健康检查脚本配置问题

### 解决方案
1. ✅ 已确认：Gateway实际运行正常，只是检查脚本权限问题
2. 🔄 待优化：重新配置健康检查脚本，使用正确权限运行
3. 🔄 待优化：考虑减少检查频率（目前5分钟一次可能过于频繁）

### 经验总结
- **监控≠运行**：监控脚本失败不代表系统故障
- **区分告警级别**：监控脚本问题不应与用户功能问题混淆
- **日志审查**：定期检查日志发现潜在问题

## 错误三：飞书云文档创建问题

### 问题描述
**第一阶段**：创建文档后只有标题，内容为空白
- 用户反馈：飞书云文档只有标题

**第二阶段**：补充内容但只有简单摘要
- 用户反馈：需要详细完整文档

**第三阶段**：追加内容时分块追加，格式不统一

### 根本原因
1. **create操作仅创建空文档**：feishu_doc create只创建标题，不会自动写入内容
2. **write操作限制**：write替换整个文档，但内容过长会导致400错误
3. **append操作限制**：append可以追加，但需要多次调用，且内容过长也会失败
4. **缺乏预先规划**：没有提前规划文档结构和内容长度

### 解决方案
1. ✅ 已实施：create后立即append内容
2. ✅ 已实施：长内容分块追加，避免单次请求过大
3. ✅ 已实施：先写摘要，再逐步补充详细内容

### 经验总结
- **先规划再执行**：创建文档前规划好结构和分块策略
- **小步快跑**：大文档分多次append，每次控制内容量
- **验证每一步**：create后立即验证，append后再次验证
- **用户确认**：每步完成后请用户确认，避免返工

### 最佳实践
```
创建大文档流程：
1. create文档（仅标题）
2. append 执行摘要
3. append 第一部分详细内容
4. append 第二部分详细内容
5. ...继续分块
6. 最后验证完整文档
```

## 错误四：GPT-5发布日期事实错误

### 问题描述
- 在深度研究报告中声称"GPT-5于2026年2月发布"
- 实际：GPT-5发布于2025年8月7日
- 2026年2月实际发布的是GPT-5.3-Codex系列

### 根本原因
1. **来源错误**：早期AI日报生成时出现了事实错误
2. **未交叉验证**：没有验证发布时间这一关键事实
3. **权威来源优先**：应优先参考Wikipedia等权威来源，而非新闻标题
4. **缺乏质疑**：对于"GPT-5刚刚发布"这种说法应产生警觉（GPT-5已发布半年）

### 解决方案
1. ✅ 已修正：更新飞书云文档，更正为GPT-5.3-Codex系列
2. ✅ 已补充：添加更正说明和准确信息
3. 🔄 待建立：建立事实核查清单

### 经验总结
- **时间敏感性事实必须验证**：发布日期、版本号等必须双重确认
- **权威来源优先**：Wikipedia > 官方博客 > 新闻报道 > 第三方分析
- **常识检查**：GPT-5发布已半年，不应是"最新消息"
- **交叉验证**：重要事实至少两个独立来源确认

### 事实核查清单
- [ ] 产品发布日期
- [ ] 版本号/型号
- [ ] 公司名称/人名拼写
- [ ] 数据来源
- [ ] 引用链接可访问

- 长内容未提前分块策略
- **改进**：大任务前先规划，拆分小步骤

- 未使用权威来源
