# Chief 发布 Skill 到共享仓库的工作流规范

更新时间：2026-03-09

## 核心原则
- 私有开发，精选发布
- 共享的是“可复用能力”，不是“整套私有环境”
- 先脱敏，再发布
- 先文档完整，再让另一方使用

## 标准流程
1. 在 Chief 私有仓库完成 skill 开发或整理
2. 检查是否适合共享：
   - 无 token / 凭据
   - 无敏感路径
   - 无仅限 Chief 私有上下文的内容
   - 对阿比确实有复用价值
3. 复制或提炼到 `agent-shared-skills` 对应目录：
   - 仅 Chief 发布、供阿比试用 → `chief/<skill-name>/`
   - 双方共同长期使用 → `shared/<skill-name>/`
4. 补齐最少结构：
   - `SKILL.md`
   - 必要脚本 / 资源
   - 在 `index.md` 中登记
   - 在 `manifest.json` 中登记
5. 标记状态：
   - 初次发布一般为 `draft`
   - 经双方验证后可升级为 `stable`
6. 发布后通知阿比侧执行 `git pull --ff-only`

## 晋升规则
- `chief/` → `shared/` 的条件：
  - 阿比已验证可用
  - 无 Chief 私有依赖
  - 文档完整
  - 至少有一个真实使用场景验证

- `abi/` → `shared/` 的条件：
  - Chief 已验证可用
  - 无阿比私有依赖
  - 文档完整
  - 至少有一个真实使用场景验证

## 不应发布的内容
- credentials
- token
- 未脱敏账号信息
- 机器专属绝对路径（如必须出现，应改成变量）
- 仅 Chief 内部使用、会污染阿比边界的技能

## 推荐同步策略
- Chief 有更新后：commit 到共享仓库
- 阿比每日定时 `git pull --ff-only`
- 如 `manifest.json` 中版本变化，阿比再按需读取对应 skill

## 一句话版
> 各自私有开发，经过脱敏和验证后，再把精选能力发布到共享仓库。
