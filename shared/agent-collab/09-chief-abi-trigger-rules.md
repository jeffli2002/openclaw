# Chief ↔ 阿比 协作触发规则

更新时间：2026-03-09
用途：定义 Chief 与阿比之间稳定、可执行、可验证的协作触发链路，避免依赖飞书群里机器人 @ 机器人。

## 一、核心结论
- 飞书群里“机器人 @ 机器人”最多适合作为**公开留痕**，不应作为**可靠触发机制**。
- 真正触发阿比执行，必须依赖以下之一：
  1. 阿比的直连可达通道（如 Open ID 定向消息）
  2. 阿比侧明确可用的私聊入口 / 会话入口
  3. OpenClaw / 多 Agent 内部路由
  4. 阿比启动时默认加载的运行记忆与规则

## 二、分层模型

### 第 1 层：公开沟通层（给老板看）
用途：
- 给 Jeff 看任务分配是否已发生
- 在群里保留协作痕迹
- 明确谁负责、谁回执、谁收口

允许做的动作：
- 在群里发任务说明
- 在群里发文档链接
- 由 Jeff 手动 @ 阿比
- 要求阿比在群里公开回执与输出结果

限制：
- 不把群消息本身当成机器人间可靠触发
- 不把“显示上 @ 成功”等同于“阿比已收到可执行事件”

### 第 2 层：执行触发层（真正让阿比干活）
优先顺序：
1. **阿比定向直连**：通过阿比 Open ID 或已验证可达的私聊目标，直接定向发送
2. **内部路由**：通过 OpenClaw / Agent runtime 内部会话或绑定路由触发
3. **人工兜底**：Jeff 在群里手动 @ 阿比，并要求其公开回执

## 三、Chief → 阿比 标准触发链路

### 标准流程
1. Chief 先判断任务类型：生活域 / 工作域 / 混合域
2. 如果任务需要阿比参与：
   - 先在群里做公开留痕（可选但推荐）
   - 再通过阿比直连通道做真正触发
3. 阿比收到后，需要在群里做**公开回执**
4. Chief 根据回执判断是否继续协作、催办或收口

### 推荐顺序
#### 场景 A：需要老板看得见
- Step 1：Chief 在群里发任务摘要
- Step 2：Jeff 手动 @ 阿比（如需要）
- Step 3：Chief 通过阿比直连通道再次定向触发
- Step 4：阿比必须回群确认

#### 场景 B：不需要群里同步，只需要执行
- Step 1：Chief 直接通过阿比直连通道触发
- Step 2：阿比在私聊或指定输出位置回执
- Step 3：必要时再同步群里

## 四、Chief 发给阿比的消息结构
每次 Chief 真正触发阿比执行时，建议统一包含：
- 任务主题
- 当前背景
- 主责归属（工作域 / 生活域 / 混合域）
- 需要阿比补充的部分
- 时间窗口 / 截止时间
- 输出要求
- 是否需要回群公开回复

## 五、阿比的回执规则
阿比被触发后，默认必须回：
1. 是否收到
2. 是否理解任务
3. 它负责哪一部分
4. 预计输出方式
5. 是否会在群里公开回复

## 六、失败与重试机制

### 判定为“未真正触发”的情况
满足任一即可：
- 只有群里消息，没有直连触发
- 群里虽然显示 @，但阿比无回执
- 使用了未验证可达的会话标识，阿比确认未收到
- 阿比没有在约定时间内回群或回私聊

### 重试顺序
1. 再次走直连通道
2. Jeff 手动 @ 阿比
3. 要求阿比只回复“收到/未收到”做链路确认
4. 若仍失败，判定该链路不可依赖，改走人工转发或内部路由
